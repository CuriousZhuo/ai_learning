# 学习记录 — 2026-05-03（模块一完成 + 模块二开始）

## 模块一已学全部内容

### 1. CPU
- 极度勤快但只会基础操作（加减乘除、搬数据）的工人
- 时钟频率 = 干活节拍，核心数 = 工人数，架构 = 干活效率
- 缓存 = CPU 的口袋（L1/L2/L3），弥补内存太慢的问题

### 2. 存储三级金字塔
- CPU 缓存（口袋，极小极快）→ 内存/书桌（中，快，断电消失）→ 硬盘/书架（大，慢，永久保存）

### 3. 操作系统
- 两大支柱：**管理资源**（进程调度、内存分配）+ **隐藏硬件**（统一接口）
- 不是"进程管理工具"（太窄）
- Windows / macOS / Linux = 三家不同的物业公司

### 4. 进程与线程
- 进程 = 资源分配的最小单位（独立内存，崩溃隔离）
- 线程 = CPU 调度的最小单位（共享内存，一崩俱崩）
- 多核使多线程真正并行

### 5. 命令行
- CLI 对操作系统直接说话 vs GUI 通过图形层间接操作

### 6. 文件系统
- 路径 = 从根走到文件的路线图；权限 = 谁能访问什么

### 7. 网络完整六步
- ① DNS 解析 ② HTTP 打包 ③ TCP 切包 ④ 路由接力 ⑤ 后端处理+查DB ⑥ 浏览器渲染
- 互联网 = 几十万个路由器接力，无中心，99% 光纤，Tier 1/2/3 但无"总路由器"

### 8. 慢网页排查表

| 症状 | 问题步骤 | 原因 |
|------|---------|------|
| "正在解析主机" | ① DNS | DNS 慢 |
| "正在建立安全连接"很久 | ②-④ | 服务器远/挂了 |
| 白屏很久突然全出来 | ⑤ 后端/DB | 处理慢 |
| 文字先出图片慢慢加载 | ⑥ | 带宽不够 |
| 直接打不开 | ②-④ | 路由故障/服务器宕 |

### 9. 前端 vs 后端
- 前端 = 用户设备上跑（界面，可见代码）
- 后端 = 远端服务器上跑（数据存取，代码不可见）
- API = 前后端之间的"点菜单"
- App 只下载前端代码；后端永远在服务器上

### 10. 数据库 vs Excel

| | Excel | 数据库 |
|------|-------|------|
| 容量 | 十几万行卡 | 几十亿行照跑 |
| 并发 | 一人用 | 几千人同时 |
| 查询 | 慢 | 毫秒级（索引） |
| 事务 | 无 | 有（全成功或全撤销） |
| 安全 | 拿文件即看 | 精确权限 |

## 模块二：博客系统架构剖析

### 博客系统三层架构

```
前端（HTML骨架 + CSS衣服 + JavaScript肌肉）
  │  API 调用（GET /api/articles, POST /api/comments, POST /api/likes）
  │  HTTP 请求/响应（JSON 格式传数据）
  ▼
后端服务器
  ├── 路由层：GET /api/articles → 查DB → 返回文章列表
  ├── 业务逻辑层：检查评论内容 → 存DB → 返回成功/失败
  └── 数据库访问层：执行 SQL
      │
      ▼
数据库
  ├── articles 表：id / title / content / author / created_at
  ├── comments 表：id / article_id / author / content / created_at
  └── likes 表：id / article_id
```

### 数据怎么在用户之间流动（关键认知修正）
- **评论者看到"发送成功"**：前端 → API → 后端 → DB → 返回响应 → 前端更新
- **朋友看到新评论**：数据库**不会主动推送**。朋友要么刷新页面（重新请求API拉新数据），要么网站用了 WebSocket 实时推送
- **纠正**：数据库只会被动应答，不是主动通知者

### JSON
- 一种所有编程语言都能读懂的通用数据格式
- `{}` = 对象，`"key": "value"` = 键值对，`[]` = 列表
- 前后端之间传数据用 JSON（人类也能大致看懂）

### API 请求协议不只 HTTP
| 协议 | 类比 | 场景 |
|------|------|------|
| HTTP/HTTPS | 写信（一信一回） | 90% 的 Web API |
| WebSocket | 打电话（持续连接，双方随时说） | 聊天、实时推送 |
| gRPC | 传真（快、格式严） | 微服务间通信 |
| GraphQL | 精确点菜（底层还是 HTTP） | 现代 App |

### 三个架构模式（知道名字即可）
- **MVC**：Model(数据库) / View(前端页面) / Controller(后端逻辑)
- **路由**：网址和功能对应（`/articles/123` → 显示第123号文章）
- **RESTful API**：统一动词操作数据（GET=拿, POST=创建, DELETE=删）

## 模块二：电商系统架构剖析

### 电商比博客多了三个核心难题
| 难题 | 技术概念 | 一句话 |
|------|---------|--------|
| 100人抢1件货 | **并发控制 + 行级锁** | 不加锁 = 同一件货卖给两个人（超卖） |
| 订单有生命周期 | **状态机** | 待付→已付→已发→已签，每跳有规则，不能乱跳 |
| 扣库存+扣款+生单需同时成功 | **数据库事务** | 三件事打包，一件失败全部回滚 |

### 行级锁防超卖（学生用自己的话正确描述）
- 无锁：两人同时读库存=1 → 两人都下单成功 → 超卖
- 有锁：先到者锁行 → 后来者等待 → 先到者改库存为0释放锁 → 后来者读到0 → 返回失败

### 电商比博客多了的数据表
users / products(stock字段是并发焦点) / orders(status字段驱动状态机) / order_items / payments

### 完整下单流程（8步）
① 开事务 → ② SELECT...FOR UPDATE锁库存行 → ③ 检查库存 → ④ 扣库存 → ⑤ 生成订单 → ⑥ 生成订单明细 → ⑦ 生成支付记录 → ⑧ 提交事务

### 博客 vs 电商核心区别
- 博客难点是**"快"**（性能/缓存）
- 电商难点是**"对"**（数据一致性/事务/锁）

## 模块二：实时协作应用架构剖析

### 核心区别：HTTP vs WebSocket
| | HTTP | WebSocket |
|------|------|---------|
| 模式 | 请求-响应（我问你答） | 全双工持久连接（拨通后不断） |
| 谁发起 | 只能客户端先开口 | **服务器也能主动开口** |
| 类比 | 寄信 | 打电话 |
| 连接 | 用完就断 | 一直连着 |
| 适用 | 博客、电商 | 聊天、实时推送、协作编辑 |

### 关键认知修正（反复出现）
- **数据库从不主动推送**。推送是后端通过 WebSocket 连接干的。
- 即时性区别不在"人数多少"，而在**通信模式**不同。

### 冲突解决（多人同时编辑）
- 最后写入胜出 / 加锁 / 操作转换（OT，Google Docs核心技术）
- 这是计算机科学最棘手的问题之一，没有完美答案

### 三个项目最终对比

| | 博客 | 电商 | 实时应用 |
|------|------|------|---------|
| 通信模式 | HTTP | HTTP | WebSocket 持久连接 |
| 核心难点 | 加载速度 | 数据一致性（并发竞态） | 消息实时性 + 冲突解决 |
| 后端角色 | 被动响应 | 被动响应 + 事务管理 | 被动响应 + 主动推送 |
| 数据库要求 | 基本 CRUD | 事务 + 行级锁 | 高频写入 + 消息持久化 |

### 学生提问：App内容推送算不算WebSocket？
- **完全算**。分两种：App内实时推送（WebSocket）和系统级通知（APNs/FCM）
- 核心原理一致：服务器主动推，客户端被动收
